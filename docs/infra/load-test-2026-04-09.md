## Diagnostico

### OK — sem problemas
- **0 erros** em todos os testes — nenhum HTTP 500, timeout ou conexao recusada
- **RAM confortavel** — pico de 43% de uso, sobram 4.5 GB
- **Nenhum container crashou** ou atingiu limit de memoria
- **Assets estaticos rapidos** — 147 RPS, ~300ms media
- **Homepage aceitavel** — ~1s com 50 users (SSR Next.js)

### Critico — requer acao

1. **Pagina `/produtos` e gargalo critico**
   - Media 8.2s com 50 users, P95 de 13.3s
   - Causa: SSR puxa todos os produtos do Medusa API em cada request
   - Impacto: usuario real esperando 8s vai embora (bounce rate alto)
   - Solucao: cache ISR ou Redis + paginacao

2. **Load average saturou CPUs**
   - Load 3.84 com 2 vCPUs = quase 2x overload
   - Maquina aguentou mas nao tem folga para picos maiores
   - Solucao: cache reduz carga; se necessario, upgrade para 4 vCPUs

3. **Medusa consumiu +32% RAM no pico**
   - Subiu de 404 MB para 536 MB (limit 1.5 GB)
   - Com mais carga pode atingir o limit
   - Solucao: connection pooling, query optimization

## Recomendacoes priorizadas

| # | Acao | Impacto esperado | Esforco |
|---|------|-----------------|---------|
| 1 | Cache de produtos (ISR/Redis) | `/produtos` de 8s → <500ms | Medio |
| 2 | Paginacao de produtos (20-30 por pagina) | Reduz payload e tempo de resposta | Baixo |
| 3 | CDN Cloudflare para assets estaticos | CSS/JS/imagens servidos do edge | Baixo |
| 4 | Cache headers no Nginx (Cache-Control) | Evita re-requests desnecessarios | Baixo |
| 5 | Upgrade para 4 vCPUs (se >50 users) | Load average confortavel | Custo mensal |

---

## Conclusao

A infraestrutura **aguenta 50 usuarios simultaneos sem cair**, com 0% de erros.
O gargalo e a pagina de produtos (SSR sem cache), que leva 8s em media.
Cache (ISR ou Redis) e a acao de maior impacto — resolveria o problema sem custo adicional de hardware.
